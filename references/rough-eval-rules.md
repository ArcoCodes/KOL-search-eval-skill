# KOL 粗估规则

> 输入：Step 2 采集到的 signals JSON（stdout 或临时文件）。
> 输出：Agent 产出粗估判断 JSON，再通过 `scripts/write_candidate.py` 写入候选池（通过=待细估，淘汰=已淘汰）

---

## 1. 前置判断字段

粗估前，Agent 先从频道的评论数据推导两个浅判断字段：

### 评论质量标记

**YouTube**（基于 `data_scrawl/youtube_data.py` 的 `comments.signals` 输出）：

| 条件 | 标记 |
|------|------|
| 模板化重复评论占比 ≥ 15% **或** 泛泛灌水占比 ≥ 40% | 严重灌水/疑似养号 |
| 泛泛灌水占比 ≥ 20% | 一般 |
| 以上都不满足 | 真实讨论 |

**TikTok / Instagram / Twitter**（基于各平台评论采样）：

| 条件 | 标记 |
|------|------|
| 重复评论数 / 总评论数 ≥ 15% | 严重灌水/疑似养号 |
| 评论内容以短句/表情/泛泛夸赞为主，缺少具体讨论 | 一般 |
| 有评论数据且有具体内容讨论 | 真实讨论 |
| 无评论数据 | 未知/待核实 |

### 受众结构（推断）

从评论语言分布推导（YouTube 取 `评论语言分布(估算)` 字段，其他平台取 `language_dist`）：

| 条件 | 推断结果 |
|------|---------|
| 英语 + 法语 + 德语 ≥ 50% | 欧美为主 |
| 日语/假名 ≥ 40% | 日本为主 |
| 韩语 ≥ 40% | 韩国为主 |
| 葡萄牙语 ≥ 40% | 巴西/葡语拉美为主 |
| 西班牙语 ≥ 40% | 拉美为主 |
| 英语 + 日语 + 韩语合计 < 25% | 非欧美为主 |
| 以上都不满足 | 分散无主导 |
| 无语言分布数据 | 未知/待核实 |

> **注意**：评论脚本输出的"英语/其他拉丁"包含西班牙语/葡萄牙语，Agent 需要结合西语/葡语单独占比来区分欧美 vs 拉美受众。如果西语/葡语占比显著（≥30%），不应简单归为"欧美为主"。

## 2. 粗估读取的 sig.json 字段

Agent 粗估不需要读 sig.json 全部内容，只看以下字段：

| 用途 | YouTube 字段路径 | 其他平台等价 |
|------|-----------------|-------------|
| 粉丝数 | `metrics.channel.subscriber_count` | `channel.followers` |
| 均播放 | `metrics.metrics.avg_views_recent` | `metrics.avg_play` / `metrics.avg_views` |
| 中位播放 | `metrics.metrics.median_views_recent` | `metrics.median_play` |
| 加权 ER | `metrics.metrics.engagement_rate_weighted` | 同名 |
| 最近更新 | `metrics.channel.latest_upload` | 从 `recent_top` 推导 |
| 逐视频播放/赞/评 | `metrics.videos[].view_count/like_count/comment_count` | `recent_top[].play/digg/comment` |
| 高播放视频播放/赞/评 | `metrics.recent_top[].views/likes/comments` | 同上 |
| 泛灌水占比 | `comments.signals.泛泛灌水占比` | `comment_signals.duplicate_comments` 等 |
| 模板化重复占比 | `comments.signals.模板化重复评论占比` | — |
| 评论语言分布 | `comments.signals.评论语言分布(估算)` | `comment_signals.language_dist` |

| 视频标题/标签 | `metrics.videos[].title`, `metrics.channel.raw_tags_top` | `recent_top[].desc` |

## 3. 注水/买量识别

粗估的核心任务是筛掉数据虚高的号。Agent 应综合以下信号做判断，不是查表对阈值。

### 互动比例失真

真实频道的赞、评、播三者由真人参与驱动，保持相对稳定的比例。注水号的播放和赞被抬高但评论跟不上，三者之间的比例就会"裂开"。

Agent 应关注：

- **点赞率异常高**：YouTube 正常频道点赞率在 2-4%，超过 8% 几乎可以确定是刷的
- **赞多评少**：评论/赞比正常在 1-3%，低于 0.5% 说明那些赞是空的——点了赞的人里没人在讨论
- **高播放低评论**：百万播放只有几十条评论不正常。注水号的典型特征是播放越高的视频评论率反而越低
- **赞评脱钩**：正常频道一条视频赞多评论也多（跨视频相关性 0.7+）；注水号因为赞是买的、评论是真的，两者各走各的

> 以上指标要看整体模式，单条视频的波动是正常的。两个以上注水信号同时命中可视为确凿证据。

### 粉播倒挂

粉丝多但播放少 → 粉丝大概率是买的：
- 5 万粉频道中位播放不到 100
- 50 万粉频道中位播放不到 1,000

### 评论灌水

评论区本身就是养的：模板化重复评论多、泛泛夸赞（"nice""great video""❤️"）密度过高、缺少对内容的具体讨论。见第 1 节评论质量标记。

### 单条爆款撑场

一条爆款占了频道 85%+ 的总播放，其余中位播放远低于均播 → 均播被严重拉高，实际影响力远低于数字表面。

### 评论语言与内容语言不一致

英文内容但评论以非英文为主 → 受众不是目标市场用户，也可能是买了非目标区的播放。

## 4. 基本面检查

除注水外，Agent 还需快速检查：

- **活跃度**：最近更新超过 90 天 → 直接淘汰
- **粉丝量**：YouTube/TikTok/Instagram < 1,000 粉、Twitter < 500 粉 → 太小，不进候选池
- **互动率（ER）**：YouTube 加权 ER < 1%、TikTok < 1.5%、Instagram < 0.5%、Twitter < 0.15% → ER 偏低，标注风险但结合其他信号判断
- **内容相关性**：频道内容与业务线完全无关（纯美食/游戏/生活vlog 等）→ 即使数据干净也应淘汰。有间接关联但不是核心受众的在依据中标注关联度

---

## 5. 判断框架

Agent 综合以上信号做出通过/淘汰判断，并给出一个 0-100 的综合评分。这是 Agent 的整体判断分，不是维度分的加总。60 分为通过线。

### 直接淘汰（出现任一）

- 互动数据注水证据确凿（多个注水信号同时命中）
- 评论质量 = "严重灌水/疑似养号"
- 粉播严重倒挂
- 不活跃（> 90 天）
- 粉丝数低于平台最低线

### 综合判断淘汰

以下信号单独出现不一定淘汰，但多项叠加应淘汰：
- 互动数据有部分异常但不够确凿
- 内容与业务线无关
- 频道主零互动 + 购买意图极低 + 评论质量一般
- ER 过低

### 通过进入候选池

- 未触发直接淘汰
- 互动数据无明显异常
- 内容与业务线有基本关联

### 标注但不否决

- 受众 = "非欧美为主" 或 "分散无主导" → 标注 "非目标区，需关注"

## 6. 粗估后的写入逻辑

| 粗估结果 | 平台明细表 | 候选池 | KOL 总表 |
|---------|-----------|--------|---------|
| **通过**（≥60 且未被直接淘汰） | **不写** | 写入，候选状态="待细估" | **不写** |
| **淘汰**（<60 或直接淘汰） | **不写** | 写入，候选状态="已淘汰(浅筛)" | **不写** |

### 去重键

| 平台 | 去重字段 |
|------|---------|
| YouTube | `主页URL` |
| TikTok | `sec_uid` |
| Instagram | `用户ID` |
| Twitter | `rest_id` |

已存在的候选池记录会更新而非重复创建。

### Agent 粗估判断 JSON

Agent 必须先把粗估结论写入 `/tmp/rough_judgment.json`：

```json
{
  "候选状态": "待细估",
  "粗估得分": 75,
  "评论质量标记": "一般",
  "受众辐射(推断)": "欧美为主",
  "粗估依据": "互动数据正常(点赞率3.2%, 赞评比2.1%, 赞评相关0.82); 5.2万粉均播1.2万, 粉播比合理; 评论有讨论但泛灌水偏高; 内容围绕AI视频生成, 与Renoise相关; 风险: 更新频率偏低(月均2条)",
  "淘汰原因": "",
  "业务线": "Renoise"
}
```

字段要求：
- `候选状态` 只允许 `待细估` 或 `已淘汰(浅筛)`
- `粗估得分` Agent 综合评分 0-100，60 分通过线
- `粗估依据` 必须写具体判断内容：互动数据是否正常及关键指标、粉播关系、评论质量、受众、内容相关性、主要风险点
- 淘汰时 `淘汰原因` 必须写清具体问题（如"点赞率10.2%+赞评比0.3%，多项注水信号命中"），不要只写"未达标"

写入候选池：

```bash
python3 scripts/write_candidate.py --from-search --signals /tmp/sig.json --judgment /tmp/rough_judgment.json --source "discovery:<keywords>" --keyword "<keywords>"
```

---

## 7. 粗估输出

Agent 用自然语言向用户汇报粗估结论，内容应涵盖：KOL 名称、平台、关键数据（粉丝/ER/均播放）、互动数据是否正常（注水风险判断）、评论质量判断、受众判断、内容相关性判断、最终结论（待细估/已淘汰）及主要理由。

---

## 8. 当前边界

- Step 3 只做浅筛，决定是否进入候选池，不做 5 维评分和报价（5 维评分和报价在 Step 4 细估阶段完成）
- Step 3 推荐把 stdout JSON 保存到 `/tmp/sig.json`，便于 Agent 审阅并传给写入脚本
- 平台明细表、KOL 总表、评估记录都在 Step 4 细估阶段再写
